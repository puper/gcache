# 架构设计

## 总览
`v2` 采用分片缓存架构：`shardedCache` 由多个 `cacheShard` 组成，每个分片独立维护：
- key -> entry 索引映射（map）
- LRU 存储（有限容量：固定数组；无限容量：可增长数组）
- 到期队列（timeWheel，24h 固定窗口单层环形时间轮）
- 分片锁（`sync.Mutex`）

## 关键设计
1. 分片并发
- 分片数必须是 2 的幂，使用按位与路由。
- 默认按 `GOMAXPROCS` 自动推导分片数，支持 Builder 覆盖。

2. LRU + TTL
- 读命中时移动到 LRU 头部。
- 写入时在有限容量模式下若分片满，则淘汰该分片尾部最久未访问项。
- 无限制容量模式下不做容量淘汰，LRU 存储按需扩容。
- TTL 使用每分片单层环形时间轮（固定 24h 窗口）。
- `Resolution` 可配置（如 1s / 1min），槽数由 `24h / resolution` 自动计算。
- 为避免极小 `Resolution` 导致超大槽数组分配，内部会将分辨率归一化到“最大槽数上限”可承受范围。
- 清理时按 `lastProcessedSlot -> currentSlot` 区间推进。
- 若 `currentSlot - start` 跨度超过窗口大小，清理仅扫描一个窗口长度，并对历史 `mappedSlot < start` 条目做兜底回收，避免长时间持锁。

3. 过期一致性
- 到期队列采用惰性删除策略：更新 TTL 时只追加新到期记录。
- 清理时通过“map 当前索引 + entry 当前 expireAt”双重校验避免误删旧记录。
- 由于时间轮是近似调度，允许在 `resolution` 粒度内延迟过期，不追求精确到纳秒。
- 长 TTL（超过 24h 窗口）在清理阶段按真实 `expireAt` 自动续挂，直到真实过期。

4. 回调策略
- 淘汰回调统一在锁外触发，避免回调重入导致锁竞争或死锁放大。

5. 稳定性约束（已完成）
- 哈希一致性：复杂泛型 key 的兜底路径必须遵循 Go 可比较语义，避免因内存布局差异导致等值 key 分片不一致。
- 回调容错：`onEvict` 属于用户代码边界，缓存内部需要做 panic 隔离，防止故障扩散。
- 32 位兼容：索引边界与容量上限计算不得依赖可能溢出的 `int` 强转。

## 复杂度（目标）
- `Get`/`Set`：平均 O(1)
- TTL 清理：近似 O(扫描槽数 + 过期候选数)
